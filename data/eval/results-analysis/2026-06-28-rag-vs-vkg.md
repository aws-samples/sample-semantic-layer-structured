# VKG vs SemanticRAG comparison

Source: `semantic_rag_vs_vkg_kmean_20260701_120646.json`.

- **SemanticRAG** = metadata query agent, layer `semantic-rag-multi_table_curated_layer-7e509128/v1` — from `metadata_query_kmean_eval_20260701_105036.json`.
- **VKG** = ontology query agent, layer `vkg-ontology_curated_layer-3d1fde76` — from `ontology_query_kmean_eval_20260701_114640.json`.
- Both k=3, same 16 GT rows + 4 multi-turn scenarios, same judges (GoalSuccess, FinalAnswerFaithfulness, SqlGrounded).

> **Enough data / no re-run needed.** Both arms are the same runs already deep-dived in `2026-06-28-rag.md` and `2026-06-28-vkg.md`, at equal k, on the identical dataset and judge set — apples-to-apples.

## Aggregate scorecard

| Metric                  | SemanticRAG     | VKG         | Winner                            |
| ----------------------- | --------------- | ----------- | --------------------------------- |
| GoalSuccess             | **0.75 ± 0.00** | 0.71 ± 0.03 | RAG (marginal, variance overlaps) |
| FinalAnswerFaithfulness | **0.75 ± 0.00** | 0.71 ± 0.03 | RAG (marginal)                    |
| SqlGrounded             | 1.00 ± 0.00     | 1.00 ± 0.00 | tie                               |
| Queries with SQL        | 9/16            | **11/16**   | VKG                               |
| Avg latency (s)         | **26.3**        | 40.9        | RAG (1.6× faster)                 |
| Avg tokens/query        | **99,749**      | 112,580     | RAG                               |

Statistically a dead heat on quality; RAG is cheaper + faster. SqlGrounded is 1.00 for both only because neither emits _ungrounded_ SQL — RAG misses are pre-SQL degrades, VKG misses are 0-row results; SqlGrounded scores vacuously on SQL-free rows (`[[reference_sqlgrounded_vacuous_pass_inflation]]`). Read GoalSuccess as the quality signal.

**11 of 16 scenarios tie at 1.0** (all 4 multi-turn + gt-05/06/08/13/14/15). The entire delta lives in 5 hard multi-hop rows, and the two architectures fail on _different_ rows — RAG +2 (gt-03, gt-07), VKG +1 (gt-02), both fail gt-01 & gt-04. They are **complementary, not dominant**: a per-row router would score ~0.88.

### Per-failure deep-dive (why each failed on a given ground truth)

### gt-row-00 — Show me policies where the insured party is also the policyholder. (RAG 0.00 · VKG 0.33)

#### Semantic-RAG

Degraded before SQL (`phase3_max_rounds`). Judge round 1: `sufficient:false`, missing `life_participant.participant_role` (presence-check `False`). No SQL emitted; advisory "data not available."

#### VKG

Emitted a self-join of two `LifeParticipant` on `holding_id`+`party_id` with `FILTER(?sk1 != ?sk2)`; translated + executed on Athena → **0 rows**. Boolean `is_deleted` compared as string (`FILTER(LCASE(STR(?del))="false")`) and bare `!=` self-join instead of `HAVING COUNT(DISTINCT participant_sk) > 1`.

#### why

The curated `life_participant` has no explicit role column. RAG refuses (demands a literal role column); VKG at least attempts the derivation ("same party on two `participant_sk`") but the boolean-as-string filter + inequality-self-join yields empty. VKG's 0.33 vs RAG's 0.00 is just that VKG produced a plausible (if empty) query. Neither used the expected discriminator (`GROUP BY … HAVING COUNT(DISTINCT participant_sk) > 1`).

### gt-row-01 — For each rider, who are the insured participants and what are their roles? (RAG 0.00 · VKG 0.00 — both fail)

#### Semantic-RAG

Degraded (`phase3_max_rounds`), no SQL. Judge demanded `life_participant.role_code` (absent), never considered `coverage.coverage_type` as the role source.

#### VKG

Emitted a 4-way join `Rider → hasRiderParticipant → RiderParticipant → hasHolding → Coverage → hasParty → Party`; executed → 0 rows. Reached Party via the `hasParty` object-property, which hits the `PARTY#` prefix asymmetry (party IRI is prefixed, `coverage.party_id` is bare), collapsing the join.

#### why

Expected SQL derives role from `coverage.coverage_type` and reaches Party by plain FK equality (`LEFT JOIN party ON c.party_id = p.party_id`). RAG's judge fixated on a non-existent `life_participant` role column; VKG chose object-property navigation over the prefixed-FK equality the mapping supports. Shared root: neither maps "participant role" to `coverage.coverage_type`; VKG adds a property-selection defect.

### gt-row-02 — List the top 5 most common party types and their human-readable descriptions. (RAG 0.00 · VKG 1.00 — **VKG win**)

#### Semantic-RAG

Degraded (`phase3_max_rounds`), no SQL. Judge: missing `party.party_type`/`party.party_id`. Slice held only party _satellites_ (`party_banking`, `party_license`, `type_codes`, `address`, …) — the base `party` dimension was never retrieved, so both columns were genuinely absent from the slice.

#### VKG

Generated a clean `SELECT ?partyType (COUNT(?p) AS ?count) … GROUP BY ?partyType ORDER BY DESC(?count) LIMIT 5`, executed → 3 rows matching expected (`Organization 7 / Trust 5 / Individual 3`).

#### why

The ontology exposes `Party/party_type` directly as a property, so VKG never had a "retrieve the right table" problem. RAG's Phase-2 retrieval favored satellites over the base `party` dim, and the slice-sufficiency judge hard-degraded. **Pure RAG slice-selection miss** — the clearest VKG advantage in the set.

### gt-row-03 — What is the total market value of all active holdings, grouped by party? (RAG 1.00 · VKG 0.00 — **RAG win**)

#### Semantic-RAG

Correct: `SUM(holding.market_value) … WHERE holding_status='Active'` joined to party via `CONCAT('PARTY#', c.party_id) = p.party_id`, grouped by party. Matched expected 2 rows.

#### VKG

Summed `HoldingSubaccount.currentvalue` (wrong metric) and filtered `Coverage.coverage_status = "Active"` (wrong entity/value domain); executed → 0 rows. The `BIND(CONCAT("PARTY#", …))` prefix bridge was actually correct — the wrong-class metric + wrong status field is the killer.

#### why

RAG grounded "market value / active" to the base `Holding` columns. VKG picked subaccount/coverage analogues — "Active" is a _holding_ status, so `coverage_status="Active"` prunes everything. **VKG wrong class/property selection.**

### gt-row-04 — For policies that have a payout schedule, show the policyholder's name, the payout frequency, and the projected next-payout amount. (RAG 0.00 · VKG 0.00 — both fail, hardest row)

#### Semantic-RAG

Degraded (`phase3_max_rounds`), no SQL. Missing `party` (base dim not retrieved — same as gt-02) plus judge mis-qualified `premium_mode` to a phantom `policy_product` table (`premium_mode` actually lives on `rider`, which was in the slice); `slice.expand` no-op'd on the chunk-less pseudo-table.

#### VKG

Emitted **no SPARQL at all** — empty answer, `degraded=null`. Phase-3 no-emit short-circuit: slice judged insufficient, round cap hit, terminal returned without an answer span (judges see "session context is empty").

#### why

The hardest row (5-table join `annuity_detail + coverage + party + policy_product + financial_activity`). RAG's dominant issue is base-dimension retrieval + judge column mis-qualification; VKG's is a control-flow gap (no answer span on the degraded terminal). Different failure modes, same unanswered row.

### gt-row-07 — Show the top 10 parties by total holding market value, including the investment product names they hold. (RAG 1.00 · VKG 0.00 — **RAG win**)

#### Semantic-RAG

Answered correctly — top-10 parties by `SUM(holding.market_value)` with product-name enrichment, 4 rows matching expected.

#### VKG

Returned a clarifying question ("Could you clarify which interpretation for 'holding', 'product', 'hold'?") instead of answering — `clarified=true`, no SPARQL. Phase-2 substring-matched the single tokens "hold"/"product" against dozens of `holding*`/`*product*` classes and columns and deemed them ambiguous.

#### why

RAG resolved the head-noun ("holding market value") to `Holding.market_value` and answered. VKG's Phase-2 term matcher over-clarified on single-token substring hits. **VKG Phase-2 over-clarify** (`[[project_phase2_phrase_overclarify]]` pattern, still present in the VKG arm).

### Ties (11/16 — reported for completeness, not deep-dived)

gt-row-05 (count parties), gt-row-06 (top-10 coverage products), gt-row-08 (monthly financial activity), gt-row-13/14/15 (advisory: metrics / what-can-I-ask / explain coverage), and all four multi-turn scenarios (mt-no-spurious-clarify, mt-parties-clarify, mt-simulated-party-count, mt-stable-options) score **1.0 on both arms**. Simple aggregates, advisory, and clarification handling are solved equally by both architectures.

---

## Synthesis & recommendations

**Failure-mode contrast:** RAG fails _before_ SQL (conservative Phase-2 retrieval + over-strict slice judge → 7/16 emit no SQL); VKG fails _after_ SQL (generates + translates + executes, but wrong property/class/join → 0 rows on 5/16 emitted). Ontop translation is **not** the bottleneck — all emitted SPARQL translated and executed cleanly (`[[project_ontop_not_the_bottleneck]]`).

1. **RAG:** force base-dimension retrieval when only `party_*` satellites match a requested `party.*` column (fixes gt-02, half of gt-04).
2. **RAG:** fix judge column→table qualification + guard dotted pseudo-tables (`premium_mode`→`rider`, not phantom `policy_product`; `[[project_slice_builder_dotted_pseudo_table]]`).
3. **VKG:** SPARQL-gen hardening — prefixed-FK equality over `hasParty` (gt-01); map "market value/active" to `Holding.*` not subaccount/coverage (gt-03); boolean audit filters as booleans (gt-00).
4. **VKG:** whole-word/phrase Phase-2 matching to stop over-clarify (gt-07); always emit an answer span on the degraded terminal (gt-04).
5. **Both:** enrichment brief should document that participant "role" is derivable from `coverage.coverage_type`, so neither hard-fails gt-01.
6. **Product:** the arms are complementary on the 5 hard rows — a per-query router would beat either standalone (~0.88 vs 0.75/0.71).

**Bottom line:** quality is a statistical tie (0.75 vs 0.71, overlapping variance) and RAG is 1.6× faster + cheaper, so **SemanticRAG is the better default today** on cost/latency at equal quality — but VKG is not dominated (it wins gt-02 outright and emits SQL on more rows). The gap is entirely 5 hard multi-hop rows with distinct, individually-fixable root causes on each side.
