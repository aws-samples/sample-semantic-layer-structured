### gt-row-00 — Show me policies where the insured party is also the policyholder.

#### expected sql

```sql
SELECT DISTINCT h.holding_id, h.policy_id, lp.party_id AS insured_party_id
FROM normalized.holding h
JOIN normalized.life_participant lp ON lp.holding_id = h.holding_id
JOIN (
  SELECT holding_id, party_id FROM normalized.life_participant
  WHERE is_deleted = false GROUP BY holding_id, party_id
  HAVING COUNT(DISTINCT participant_sk) > 1
) dup ON dup.holding_id = h.holding_id AND dup.party_id = lp.party_id
WHERE lp.is_deleted = false AND h.is_deleted = false
```

#### expected answer

20 rows, e.g. `POLICY#POL00000011 / POL00000011 / PARTY000011`.

#### observed sql

_None — degraded before SQL generation._

#### observed answer

"I found relevant tables but the data needed to answer this question isn't available in this semantic layer. Missing: life_participant.participant_role (or equivalent role/type column identifying Insured vs Owner/Policyholder for each participant_sk). This usually means the column or lookup doesn't exist in the underlying schema…" (`degraded=phase3_max_rounds`, 0 rows)

#### error analysis

**Genuine data gap.** Judge round 1: `sufficient:false`, missing `life_participant.participant_role`, presence-check `False`. The slice contained 16 tables including `life_participant`, whose columns are `holding_id, party_id, participant_sk, issue_age, issue_gender, underwriting_class, tobacco_premium_basis` + audit cols — **no role/type discriminator exists**. `participant_role` appears in the logs only inside the judge's own missing-claim string. The slice enrichment prose _claims_ "life_participant carries the participant's role," but no such column is materialized, so the judge was correct to reject. Note the VKG agent answered this same row from the _same_ schema by treating "same party, two `participant_sk`" as the discriminator (via self-join) — i.e. the question is answerable without an explicit role column, but the RAG judge insists on a literal role column that doesn't exist.

#### potential fix

Two options: (a) fix the enrichment brief so it no longer promises a `participant_role`/role column that the schema doesn't expose — and instead documents the derivation recipe (same `party_id` with >1 distinct `participant_sk` on a holding = party plays multiple roles), so the judge accepts a derivable answer; or (b) if a role column genuinely belongs in the curated layer, add it upstream. Option (a) is the cheaper alignment fix and matches how VKG reached the answer.

### gt-row-02 — List the top 5 most common party types and their human-readable descriptions.

#### expected sql

```sql
SELECT p.party_type, COUNT(*) AS party_count
FROM normalized.party p
WHERE p.is_deleted = false
GROUP BY p.party_type ORDER BY party_count DESC LIMIT 5
```

#### expected answer

3 rows: `Organization / 7`, `Trust / 5`, `Individual / 3`.

#### observed sql

_None — degraded before SQL generation._

#### observed answer

"…Missing: normalized.party.party_type, normalized.party.party_id. This usually means the column or lookup doesn't exist in the underlying schema…" (`degraded=phase3_max_rounds`, 0 rows)

#### error analysis

**Slice-selection miss (Phase 2), presenting as apparent judge over-rejection.** Judge round 1: `missing = {'party.party_type': False, 'party.party_id': True}`. The slice held 6 tables — `address, distribution_level, life_participant, party_banking, party_license, type_codes` — but **the base `party` dimension is NOT among them**. Both columns are real in the schema (the slice's own recipes even reference `party.party_type` as the GROUP BY label), but they live on `party`, which Phase 2 never retrieved. The `party_id: True` presence hit is a _false match_ on satellite tables (`party_banking.party_id`, `party_license.party_id`, etc.), not the real `party` dim. So the judge correctly saw `party.party_type` absent from the slice — the deeper cause is retrieval, which grabbed party _satellites_ instead of the base dimension. (This trace also shows a `SliceSufficiency structured output validation failed` retry — the judge round cost two model calls.)

#### potential fix

Phase-2 retrieval must include the base `party` dimension whenever party attributes are requested, rather than only its satellites (`party_banking`/`party_license`/`type_codes`). Boost the base-entity table in the retrieval ranking, or add a rule: if a `party.*` column is requested and only `party_*` satellites are in the slice, expand to fetch `party`. This is the most actionable fix — it also directly helps gt-row-04.

### gt-row-04 — For policies that have a payout schedule, show the policyholder's name, the payout frequency, and the projected next-payout amount.

#### expected sql

```sql
SELECT p.full_name AS policyholder_name,
       COALESCE(pp.premium_mode, c.product_code) AS payout_frequency,
       fa.transaction_amount AS projected_next_payout_amount,
       fa.effective_date AS payout_effective_date, fa.activity_type AS payout_type,
       ad.holding_id AS policy_id
FROM normalized.annuity_detail ad
JOIN normalized.coverage c ON c.holding_id = ad.holding_id AND c.is_deleted = false
JOIN normalized.party p ON p.party_id = CONCAT('PARTY#', c.party_id) AND p.is_deleted = false
LEFT JOIN normalized.policy_product pp ON pp.product_code = c.product_code AND pp.is_deleted = false
LEFT JOIN normalized.financial_activity fa ON fa.holding_id = ad.holding_id
     AND fa.activity_type IN ('Withdrawal','Dividend','Claim') AND fa.is_deleted = false
WHERE ad.deleted = false ORDER BY ad.holding_id, fa.effective_date DESC LIMIT 10
```

#### expected answer

5 rows, e.g. `Scott Neil Davis / VUL / 35512.98 / Dividend / POLICY#POL00000011`.

#### observed sql

_None — degraded before SQL generation._

#### observed answer

"…Missing: party, policy_product.premium_mode. This usually means the column or lookup doesn't exist in the underlying schema…" (`degraded=phase3_max_rounds`, 0 rows)

#### error analysis

**Mixed: slice-selection miss + judge mis-qualification.** Slice had 14 tables (`holding, holding_payout, holding_projection, rider, rider_participant, coverage, financial_activity, annuity_detail`, …) but **no `party` table and no `policy_product` table**. Two distinct issues:

- `party` → **genuine slice miss** (policyholder name needs `party`, which was not selected — same Phase-2 gap as gt-row-02).
- `policy_product.premium_mode` → **judge mis-qualification**: `premium_mode` IS a real column, but it lives on `normalized.rider` (already in the slice, described as "Frequency the rider premium is billed. Observed values: Monthly, Quarterly, Annual"), not on `policy_product` (which isn't even a table in this layer). The judge attached the column to a phantom `policy_product` table; the `.premium_mode' has no chunk` log line is the dotted-pseudo-table symptom — the wrong `table.column` string resolved to a phantom table with no chunk, so `slice.expand` was a no-op and the run short-circuited. The payout frequency was in fact answerable from `rider.premium_mode` already in the slice.

#### potential fix

(1) Same Phase-2 fix as gt-row-02 — pull the base `party` table when policyholder name is requested. (2) Fix the judge's column→table qualification so it binds `premium_mode` to `rider` (where it exists) rather than inventing `policy_product.premium_mode`; guard the slice-expand path against dotted pseudo-tables that resolve to no chunk (the known slice-builder dotted-pseudo-table bug).

### Cross-cutting patterns (RAG)

1. **Phase-2 drops the base `party` dimension** (rows 02, 04): retrieval favors party _satellites_ (`party_banking`, `party_license`, `type_codes`) over the base `party` table, so party attributes look "missing." Highest-leverage fix — boost/force base-dimension retrieval when its columns are requested.
2. **Enrichment brief promises a participant-role column that the schema lacks** (rows 00, 01): the curated `life_participant` has no `participant_role`/`role_code`; "role" is derivable from `coverage.coverage_type` or from multi-`participant_sk` self-joins. Fix the brief to document the derivation recipe so the judge accepts a derivable answer instead of hard-degrading.
3. **Judge column mis-qualification / dotted pseudo-table** (row 04): the judge attaches a real column to the wrong (sometimes phantom) table (`policy_product.premium_mode` vs the real `rider.premium_mode`), and `slice.expand` no-ops on the phantom, forcing degrade.

Contrast with VKG: the VKG agent answered gt-row-00 and gt-row-01 from the _same_ underlying schema (deriving role via self-join / coverage_type) where the RAG judge hard-degraded — the RAG failures are dominated by conservative Phase-2 retrieval + an over-strict slice-sufficiency judge, not by genuine absence of data (except the literal role column). SqlGrounded stays 1.00 because no ungrounded SQL was ever emitted — the failures are all pre-SQL degrades, which GoalSuccess/FAF penalize.
