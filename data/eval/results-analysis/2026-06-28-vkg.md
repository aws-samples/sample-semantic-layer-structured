### gt-row-00 — Show me policies where the insured party is also the policyholder.

#### expected sql

```sql
SELECT DISTINCT h.holding_id, h.policy_id, lp.party_id AS insured_party_id
FROM normalized.holding h
JOIN normalized.life_participant lp ON lp.holding_id = h.holding_id
JOIN (
  SELECT holding_id, party_id
  FROM normalized.life_participant
  WHERE is_deleted = false
  GROUP BY holding_id, party_id
  HAVING COUNT(DISTINCT participant_sk) > 1
) dup ON dup.holding_id = h.holding_id AND dup.party_id = lp.party_id
WHERE lp.is_deleted = false AND h.is_deleted = false
```

#### expected answer

20 rows, e.g. `POLICY#POL00000011 / POL00000011 / PARTY000011`; `POL00000010 / PARTY000010`; `POL00000012 / PARTY000012`.

#### observed sql

```sparql
SELECT DISTINCT ?holding_id WHERE {
  ?lp1 a :LifeParticipant ; :LifeParticipant/holding_id ?holding_id ;
       :LifeParticipant/party_id ?party_id ; :LifeParticipant/participant_sk ?sk1 ;
       :LifeParticipant/is_deleted ?del1 .
  ?lp2 a :LifeParticipant ; :LifeParticipant/holding_id ?holding_id ;
       :LifeParticipant/party_id ?party_id ; :LifeParticipant/participant_sk ?sk2 ;
       :LifeParticipant/is_deleted ?del2 .
  FILTER(?sk1 != ?sk2)
  FILTER(LCASE(STR(?del1)) = "false")
  FILTER(LCASE(STR(?del2)) = "false")
} ORDER BY ?holding_id
```

(SqlGrounded judge saw the translated Athena result headers `HOLDING_ID, IS_DELETED, PARTICIPANT_SK, PARTY_ID` → confirms translate+execute happened.)

#### observed answer

"The query returned no rows, so no policies matching 'insured also policyholder' were found … this may reflect the join/filter conditions used rather than confirming that no such policies exist." (0 rows)

#### error analysis

SPARQL-generation defect, not an Ontop-engine or data defect. The intent (same `party_id` bound to two distinct `participant_sk` on the same `holding_id`) is semantically right, but two things sink it: (1) the `is_deleted` discriminator is a **typed boolean** in the curated schema, and the agent compares it as a string via `FILTER(LCASE(STR(?del)) = "false")` — a boolean-vs-string literal mismatch that the expected SQL avoids with `WHERE is_deleted = false`; (2) the plain `FILTER(?sk1 != ?sk2)` self-join projecting only `?holding_id` behaves differently from the expected `GROUP BY holding_id, party_id HAVING COUNT(DISTINCT participant_sk) > 1` when Ontop materializes the two triple-pattern copies. Net: empty inequality join → 0 rows.

#### potential fix

Harden the SPARQL generator to compare boolean audit columns as booleans (`FILTER(?del1 = false)` or omit the STR() coercion), and prefer a `GROUP BY … HAVING COUNT(DISTINCT …) > 1` shape for "same X plays two roles" self-joins rather than a bare `!=` self-join.

### gt-row-01 — For each rider, who are the insured participants and what are their roles?

#### expected sql

```sql
SELECT r.rider_id, r.rider_name, r.rider_code, r.rider_type, r.rider_status, r.holding_id,
       c.coverage_id, c.coverage_type AS participant_role, c.party_id,
       COALESCE(NULLIF(p.full_name,''), CONCAT(p.first_name,' ',p.last_name)) AS participant_name,
       p.party_type, c.issue_age, c.issue_gender, c.underwriting_class
FROM normalized.rider r
JOIN normalized.coverage c ON r.holding_id = c.holding_id
LEFT JOIN normalized.party p ON c.party_id = p.party_id
WHERE r.is_deleted = false AND c.is_deleted = false
ORDER BY r.rider_id, c.coverage_id LIMIT 10
```

#### expected answer

10 rows, e.g. `RIDER00258816 / Accidental Death Benefit / RDR3763 / role=Rider / PARTY000057 / underwriting_class=Preferred Plus`.

#### observed sql

```sparql
SELECT DISTINCT ?riderName ?fullName ?coverageType WHERE {
  ?rider a :Rider ; :Rider/rider_name ?riderName ; :Rider/is_deleted ?riderDeleted ;
         :Rider/hasRiderParticipant ?rp . FILTER(LCASE(STR(?riderDeleted))="false")
  ?rp a :RiderParticipant ; :RiderParticipant/is_deleted ?rpDeleted ;
      :RiderParticipant/hasHolding ?holding . FILTER(LCASE(STR(?rpDeleted))="false")
  ?coverage a :Coverage ; :Coverage/hasHolding ?holding ; :Coverage/hasParty ?party ;
            :Coverage/coverage_type ?coverageType ; :Coverage/is_deleted ?covDeleted . FILTER(LCASE(STR(?covDeleted))="false")
  ?party a :Party ; :Party/full_name ?fullName .
} ORDER BY ?riderName ?fullName
```

#### observed answer

"The query returned no rows, so no rider-participant-role combinations were found … this may reflect the filters or join logic used rather than confirming an absence of such data." (0 rows)

#### error analysis

Wrong join-property selection (SPARQL-gen defect). The expected query reaches Party with a plain FK equality `LEFT JOIN party ON c.party_id = p.party_id` and derives the "role" from `coverage.coverage_type`. The agent instead navigated `Coverage/hasParty` (object property) and chained `Rider → hasRiderParticipant → RiderParticipant → hasHolding → Coverage`. The `hasParty` object property does not resolve through the Ontop mapping the way the datatype-FK does — Party's IRI carries a `PARTY#` prefix while `coverage.party_id` is the bare key (the recurring party↔coverage prefix asymmetry), so the object-property join collapses to empty. The extra `RiderParticipant` hop is also unnecessary (expected joins Rider→Coverage directly on `holding_id`).

#### potential fix

Bias the generator to reach Party via the explicit `party_id`/`PARTY#`-prefixed FK equality (or add a working `hasParty` mapping that applies the prefix), and prefer the shortest `Rider→Coverage on holding_id` path instead of routing through `RiderParticipant`.

### gt-row-03 — What is the total market value of all active holdings, grouped by party?

#### expected sql

```sql
SELECT p.party_id, p.full_name AS party_name,
       COUNT(DISTINCT h.holding_id) AS holding_count, SUM(h.market_value) AS total_market_value
FROM normalized.holding h
JOIN normalized.coverage c ON h.holding_id = c.holding_id
JOIN normalized.party p ON CONCAT('PARTY#', c.party_id) = p.party_id
WHERE h.holding_status = 'Active' AND h.is_deleted = false
  AND c.is_deleted = false AND p.is_deleted = false
GROUP BY p.party_id, p.full_name ORDER BY total_market_value DESC LIMIT 10
```

#### expected answer

2 rows: `Thomas Jill Davis / 317902.53`; `Jared Xavier Parker / 90066.59`.

#### observed sql

```sparql
SELECT ?full_name (SUM(xsd:decimal(?currentvalue)) AS ?totalMarketValue) WHERE {
  ?hs a :HoldingSubaccount ; :HoldingSubaccount/holding_id ?holdingId ;
      :HoldingSubaccount/currentvalue ?currentvalue ; :HoldingSubaccount/deleted ?deleted . FILTER(LCASE(STR(?deleted))="false")
  ?cov a :Coverage ; :Coverage/holding_id ?holdingId ; :Coverage/coverage_status ?covStatus ;
       :Coverage/party_id ?covPartyId . FILTER(?covStatus = "Active")
  ?party a :Party ; :Party/party_id ?partyId ; :Party/full_name ?full_name .
  BIND(CONCAT("PARTY#", ?covPartyId) AS ?partyPrefixed) FILTER(?partyPrefixed = ?partyId)
} GROUP BY ?full_name
```

#### observed answer

"The query returned no rows, so no total market value by party could be retrieved … this may reflect the join/filter conditions used rather than an actual absence of active holdings." (0 rows)

#### error analysis

Wrong class/property selection plus a status-literal filter on the wrong entity. The metric "market value of active holdings" lives on `holding.market_value` gated by `holding.holding_status = 'Active'`. The agent summed `HoldingSubaccount.currentvalue` (a subaccount-level figure, different concept) and gated on `Coverage.coverage_status = "Active"`. Coverage status is a different value domain — "Active" is a _holding_ status, so `coverage_status="Active"` prunes every row → empty result. (The `BIND(CONCAT("PARTY#", ?covPartyId))` prefix bridge here is actually correct and matches the expected `CONCAT('PARTY#', c.party_id)`, so the prefix is not the killer this time — the wrong-class metric + wrong status field is.)

#### potential fix

Improve Phase-2/slice grounding so "market value of holdings" maps to `Holding.market_value` and "active" maps to `Holding.holding_status`, not the subaccount/coverage analogues. A property-disambiguation hint ("market_value is a Holding property; 'Active' is a holding_status value") in the ontology annotations would steer generation.

### gt-row-07 — Show the top 10 parties by total holding market value, including the investment product names they hold.

#### expected sql

```sql
SELECT p.party_id, p.full_name AS party_name, SUM(h.market_value) AS total_market_value,
  ARRAY_JOIN(ARRAY_AGG(DISTINCT CASE WHEN hs.fundname IS NOT NULL THEN hs.fundname
     WHEN pp.product_name IS NOT NULL THEN pp.product_name ELSE 'N/A' END), ', ') AS investment_products
FROM normalized.party p
JOIN normalized.coverage c ON c.party_id = REPLACE(p.party_id,'PARTY#','') AND c.is_deleted = false
JOIN normalized.holding h ON h.holding_id = c.holding_id AND h.is_deleted = false
LEFT JOIN normalized.holding_subaccount hs ON hs.holding_id = h.holding_id AND hs.deleted = false
LEFT JOIN normalized.policy_product pp ON pp.product_code = c.product_code AND pp.is_deleted = false
WHERE p.is_deleted = false GROUP BY p.party_id, p.full_name ORDER BY total_market_value DESC LIMIT 10
```

#### expected answer

4 rows, e.g. `Scott Neil Davis / 651373.05 / "N/A, Universal Life"`; `Thomas Jill Davis / 317902.53 / "20-Year Term Life"`.

#### observed sql

_None — agent returned a clarifying question instead._

#### observed answer

"Could you clarify which interpretation you mean for: 'holding', 'product', 'hold'?" (`clarified=true`, no SPARQL)

#### error analysis

Over-clarification from Phase-2 substring over-matching. The 06-27 log line for the identical defect shows the resolver expanding the single tokens "product" and "hold" into a huge candidate list — `investproduct, policyproduct, product_name, holdingsubaccount, holdingpayout, holding, holdingactivity, holdingarrangement, holdingdbg, holdingloan, holdingprojection, holdingrestriction, holding_number, holding_status, holding_type, holdingid, holding_id …`. Because every `holding*`/`*product*` class and column substring-matched, the term was deemed ambiguous and the agent asked to disambiguate instead of picking the obvious `holding.market_value` + product-name enrichment. This is the known Phase-2-phrase over-clarify pattern.

#### potential fix

Require multi-token phrase / whole-word matching before triggering clarify (the same fix landed for the RAG agent), and suppress clarify when a dominant head-noun mapping exists ("holding market value" → `Holding.market_value`). Single-token substring hits against dozens of same-family names should not, by themselves, force a clarify.

### Cross-cutting patterns (VKG)

1. **Party↔Coverage prefix asymmetry** (rows 01, 03, 04, 07 all touch it): `party.party_id` = `PARTY#PARTYnnnnnn` while `coverage.party_id` = bare `PARTYnnnnnn`. Expected SQL always bridges with `CONCAT('PARTY#', c.party_id) = p.party_id` / `REPLACE(...)`. When the agent navigates via the `hasParty` object-property (row 01) instead of the explicit prefixed-FK equality, the mapping join collapses to empty. This is SPARQL-property-selection, **not** an ontop-translate-engine defect.
2. **Boolean audit columns compared as strings** (row 00): `FILTER(LCASE(STR(?del))="false")` vs the typed `is_deleted = false`.
3. **Wrong-class metric selection** (row 03): subaccount/coverage analogues chosen over the base `Holding.market_value` / `holding_status`.
4. **Phase-2 over-clarify** (row 07) and **Phase-3 no-emit short-circuit** (row 04) are graph-control defects, not translate defects.

None of the five failures are ontop-translate-engine defects — consistent with the prior finding that Ontop reformulates all well-formed multi-hop queries. The remaining gap is SPARQL-generation quality (property/class/literal selection) plus graph-control (over-clarify, no-emit terminal). SqlGrounded stays 1.00 because every emitted SPARQL translated and executed cleanly; the failures are semantic-emptiness (0 rows) and non-emission, which GoalSuccess/FAF catch.
